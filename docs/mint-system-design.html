Mint — System Design

A Go service that issues, validates, and meters API keys for other backends. Every incoming request to a customer's service triggers one POST /v1/validate — a single Redis round-trip on a cache hit — that checks key + rate limit + quota. Postgres is the source of truth and off the hot path entirely. Built, measured, and shown with the real code.

0. Orientation

Two pictures to anchor everything below: who talks to whom, and the journey of a single /validate request.

System context — who talks to whom

Other backends adopt Mint by wrapping their router in one line of the keyservice-go SDK. From then on, every request they receive triggers a /validate to Mint, which leans on Redis for the hot path and Postgres as the durable source of truth.

Request lifecycle — a single /validate

The branch is the whole performance story: a cache hit answers from RAM (L1) or one Redis hop (L2) with no Postgres at all; a miss falls through to a single indexed Postgres lookup, then writes the answer back into both cache tiers so the next identical key is warm. A valid key then passes through one Redis Lua call that does rate-limit + quota + metering together.

1. Requirements

Concept · functional vs non-functional

Functional (FR) = what the system does (a feature). Non-functional (NFR) = how well — latency, throughput, availability, security. The NFRs are what shape this architecture: anyone can validate a key; doing it 5,000×/second under 20 ms is the hard part.

Functional requirements (MoSCoW)

MUST The core — the product is meaningless without these
  1. Issue keys per tenant — show the plaintext once, store only a hash.
  2. Validate a key on the hot path: POST /v1/validate → key valid? tenant active?
  3. Rate-limit per key (burst-tolerant) and meter + quota per tenant per month.
  4. Revoke a key and have every replica honour it within milliseconds.
SHOULD What makes it production-grade
  1. Cache the hot path so Postgres is off it entirely (L1 + L2 + invalidation).
  2. Observability: RED metrics, structured logs, readiness, dashboards.
  3. A drop-in client SDK so a backend integrates in one line.
WON'T Out of scope now — named, so it's a decision
  1. Key rotation UI, multi-region, per-endpoint scopes/RBAC, a hosted control-plane dashboard.

Non-functional requirements & SLOs

Concept · SLO & fail-mode asymmetry

An NFR must be a number. Mint's defining design rule is an asymmetric failure mode: auth fails closed (a doubt → reject; security > availability) while rate-limiting fails open (Redis down → allow; availability > strictness). Same kind of failure, opposite choice, decided by what's at stake.

The core trade-offs

Decision 1 Key hashing — what runs on every request
Argon2id / bcrypt — memory-hard, deliberately slow. Correct for human passwords.
HMAC-SHA256 + server-side pepper — microsecond-cheap, keyed by a secret held outside the DB.
We pick HMAC + pepper. Hashing runs on the 5,000 rps hot path, so a deliberately-slow KDF would blow the p99 < 20 ms budget. API keys are already high-entropy (≈190 bits) — they don't need the brute-force resistance Argon2id buys for low-entropy passwords. The pepper (an env-var secret, not in Postgres) means a DB-only dump can't be brute-forced offline, and because the hash is deterministic it doubles as the key_hash BYTEA UNIQUE lookup index — one indexed SELECT, no per-row salt to fetch-then-compare.
Decision 2 Where do rate-limit + quota + metering run
In Go, per replica — but each replica has its own memory, so counts diverge and a key could exceed its limit ×replicas.
Three separate Redis calls — correct shared state, but 3 round-trips and 3 races on the hot path.
One atomic Redis Lua script — refill the bucket, check rate, check quota, INCR usage — all in a single round-trip.
We pick one Lua call. A read-modify-write across replicas needs Redis-side atomicity that Go locks can't give. Bundling all three into one EVALSHA collapses three races and three round-trips into one — the quota check and the usage INCR ride the round-trip the token bucket already pays. Zero added hot-path latency for two extra features.
Decision 3 Usage metering — synchronous or coalesced
Write each request to Postgres — accurate, but ~5,000 writes/s of contention on the money table.
Redis INCR on the hot path + background flusher — count in Redis, mirror to Postgres every interval.
We pick coalescing. The hot path does zero Postgres writes; usage lives as a Redis counter and a background flusher batches it. The Postgres write rate becomes active_tenants / flush_interval — invariant to load. Measured: ~10 writes/s instead of thousands (a 376×–500× reduction). The flusher's UPSERT mirrors the absolute value, so it's idempotent — a double-flush can't corrupt the count.

2. Capacity estimates & measured numbers

Concept · Little's Law

Items in flight = arrival rate × time each spends inside: L = λ × W. For Mint at the 5,000 rps target with a warm-cache service time of ~2 ms, L = 5000 × 0.002 = 10 in-flight requests. The hot path is so short that concurrency is trivially small — the engineering isn't holding connections, it's keeping W tiny by never touching Postgres on a hit.

Cache hit rate is an outcome, not a dial

With a 5-minute TTL and a working set of U unique keys over N requests, each unique key causes ~one cold miss and is then warm: hit_rate ≈ 1 − U/N. That model predicted, and the load test confirmed, 99.7% at 1,000 keys and 96.7% at 10,000 keys over 300k requests — the misses are almost exactly one per unique key. The hit rate is a property of the workload's repetition, which is why it's measured, not claimed.

Write-amplification — the number the flusher kills

Naïve per-request metering writes to Postgres at the request rate. Coalescing makes the durable write rate active_tenants / flush_interval — at 300 tenants and a 30 s interval, ~10 writes/s no matter how much traffic arrives. That decoupling is the whole point of Section 1's Decision 3.

Measured (the honest numbers)

Hardware: MacBook Pro, Apple M2 Pro (10 cores), 16 GB. The load generator runs on the same machine as the full stack, so absolute throughput is a floor, not a ceiling — but the before/after deltas are reliable. Every number is reproducible from a committed script in benchmarks/.

Deployment — small, static, ordered

A multi-stage build emits a static, symbol-stripped binary into a bare Alpine image running as a non-root user — ~22 MB versus ~800 MB single-stage. Manifests are copied before source so the dependency layer caches independently.

keyservice/Dockerfile · multi-stage, ~22 MB
FROM golang:1.25-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download                       # cached unless deps change
COPY . ./
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /out/keyservice .

FROM alpine:3.20
RUN apk add --no-cache ca-certificates
RUN adduser -D -u 10001 appuser           # run unprivileged, not root
USER appuser
COPY --from=builder /out/keyservice /usr/local/bin/keyservice
EXPOSE 8080
ENTRYPOINT ["/usr/local/bin/keyservice"]

nginx is the only published port. The keepalive 32 pool only works because HTTP/1.1 is forced and the default Connection: close header is cleared — otherwise every request would open a fresh upstream socket.

infra/nginx.conf · upstream keep-alive to a round-robined replica set
worker_processes auto;                    # one worker per CPU core

http {

    server keyservice:8080;               # Docker DNS → all replicas, round-robin
    keepalive 32;                         # idle-connection pool per worker
  }

    listen 80;
    location / {
      proxy_pass http://keyservice_backend;
      proxy_http_version 1.1;             # required for keep-alive…

4. API contract & data model

The hot-path contract

One endpoint carries all the load. The key rides in a standard Bearer header; the response is a tiny JSON verdict. Status codes are carefully distinct so a caller can tell wrong key from too many requests.

POST /v1/validate
Authorization: Bearer ak_live_8sK2pX9q...

200 OK    { "valid": true, "tenant_id": "…", "key_id": "…" }   # served + metered
401       { "valid": false }                                    # unknown / revoked / suspended — no reason leaked
429       { "valid": true, "error": "rate limit exceeded" }     # valid key, over burst
429       { "valid": true, "error": "monthly quota exceeded" }  # valid key, over quota
StatusWhenPolicy
200valid key, within limitsserved, usage incremented
400missing / malformed Bearer headerreject at parse
401unknown, revoked, or suspended tenantfail closed, quiet
429over rate limit OR over monthly quotadistinct error strings; 429 ≠ 401
500real Postgres errornot cached

Admin endpoints (create tenant, issue key, revoke key, read usage) sit behind an X-Admin-Token header compared in constant time.

Data model

A small normalised schema in Postgres. api_keys → tenants is ON DELETE RESTRICT (you can't delete a tenant that still holds keys — protects the audit trail); usage_counters → tenants is ON DELETE CASCADE (usage is derived data, safe to drop). The hashed key column is UNIQUE — that constraint is the validate lookup index.

keyservice/migrations/0002_create_api_keys.up.sql
CREATE TABLE api_keys (
    id            UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id     UUID        NOT NULL REFERENCES tenants(id) ON DELETE RESTRICT,
    name          TEXT        NOT NULL,
    key_prefix    TEXT        NOT NULL,
    key_hash      BYTEA       NOT NULL UNIQUE,          -- the /validate lookup index
    status        TEXT        NOT NULL DEFAULT 'active'
                              CHECK (status IN ('active', 'revoked')),
    last_used_at  TIMESTAMPTZ NULL,
    revoked_at    TIMESTAMPTZ NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_api_keys_tenant_id ON api_keys (tenant_id);

Validation is a single JOIN requiring status='active' on both tables, so a revoked key or a suspended tenant both collapse to zero rows — no fetch-then-check, no second query, no TOCTOU window. Every miss returns the one sentinel ErrKeyNotValid, so nothing leaks about why.

keyservice/internal/store/store.go · ValidateKey — one JOIN, fail-closed
func (s *Store) ValidateKey(ctx context.Context, keyHash []byte) (ValidatedKey, error) {
	var vk ValidatedKey
	err := s.pool.QueryRow(ctx,
		`SELECT k.id, k.tenant_id, COALESCE(t.monthly_quota,0)
		FROM api_keys k
		JOIN tenants t ON t.id = k.tenant_id
		WHERE k.key_hash = $1
		AND k.status = 'active'
		AND t.status = 'active'`, keyHash).Scan(&vk.KeyID, &vk.TenantID, &vk.MonthlyQuota)
	if err != nil {
		if errors.Is(err, pgx.ErrNoRows) {
			return ValidatedKey{}, ErrKeyNotValid      // any miss → one quiet sentinel
		}
		return ValidatedKey{}, err
	}
	return vk, nil
}

The quota rides in the same SELECT (COALESCE(monthly_quota,0)), so it travels into the cache entry and the per-request quota check needs no extra lookup.

5. Component deep dives

5.1 Keys — unguessable, cheap, hash-only

Keys are drawn from the OS CSPRNG with rejection sampling to avoid modulo bias (a raw byte is 0–255; 256 isn't a multiple of 62, so the low residues would be slightly over-represented — dropping bytes ≥ 248 fixes it). 32 base62 chars ≈ 190 bits of entropy. Only the HMAC digest is stored.

keyservice/internal/keys/keys.go
func Generate() (string, error) {
	const bodyLen = 32
	const maxByte = 62 * 4 // 248 — largest multiple of 62 that fits in a byte
	out := make([]byte, bodyLen)
	buf := make([]byte, 1)
	for i := 0; i < bodyLen; {
		if _, err := rand.Read(buf); err != nil {       // crypto/rand — OS CSPRNG
			return "", err
		}
		if buf[0] >= maxByte {                            // rejection sampling
			continue
		}
		out[i] = alphabet[buf[0]%62]
		i++
	}
	return "ak_live_" + string(out), nil
}

// Hash returns HMAC-SHA256(pepper, key) — 32 raw bytes for the BYTEA column.
func Hash(pepper, key string) []byte {
	mac := hmac.New(sha256.New, []byte(pepper))
	mac.Write([]byte(key))
	return mac.Sum(nil)
}

5.2 The cache — L1 + L2 + pub/sub invalidation

Concept · multi-level read-through cache

L1 is a per-replica Go map (RAM, ~0.2 ms, no network). L2 is shared Redis (absorbs cross-replica and cold-start misses). Postgres is the last resort. A miss writes the answer back through both tiers (write-through); an L2 hit backfills L1 so the next read stays in-process.

L1 is a map guarded by a sync.RWMutex with lazy per-entry expiry — readers take the shared lock, expiry is checked on read rather than by a background sweeper, keeping the hot path lock-light.

keyservice/internal/cache/cache.go · L1 in-process cache
type Cache struct {
	mu   sync.RWMutex
	data map[string]entry
}
func (c *Cache) Get(key string) (any, bool) {
	c.mu.RLock()
	e, ok := c.data[key]
	c.mu.RUnlock()
	if !ok || time.Now().After(e.expiresAt) {     // lazy TTL: expired = miss
		return nil, false
	}
	return e.value, true
}
func (c *Cache) Set(key string, value any, ttl time.Duration) {
	c.mu.Lock()
	c.data[key] = entry{value: value, expiresAt: time.Now().Add(ttl)}
	c.mu.Unlock()
}

The tiered read lives in the validate handler. Negative caching stores invalid verdicts too (short 30 s TTL vs 5 min for valid) so a bad-key flood can't hammer Postgres — but a real DB error is never cached.

keyservice/internal/api/api.go · the three-tier read in /validate
// --- L1: in-process ---
if v, hit := s.cache.Get(cacheKey); hit {
	s.l1Hits.Add(1)
	s.writeValidateResult(w, r, cacheKey, v.(cache.Result))
	return
}
// --- L2: Redis ---
if res, hit := s.l2.Get(ctx, cacheKey); hit {
	s.l2Hits.Add(1)
	s.cache.Set(cacheKey, res, ttlFor(res))        // backfill L1
	s.writeValidateResult(w, r, cacheKey, res)
	return
}
// --- L3: Postgres (source of truth) ---
s.misses.Add(1)
vk, err := s.store.ValidateKey(ctx, keyHash)
var res cache.Result
if err != nil {
	if !errors.Is(err, store.ErrKeyNotValid) {
		writeJSONError(w, http.StatusInternalServerError, "internal error")
		return                                       // real DB error → don't cache
	}
	res = cache.Result{Valid: false}                 // negative cache (short TTL)
} else {
	res = cache.Result{Valid: true, TenantID: vk.TenantID, KeyID: vk.KeyID, MonthlyQuota: vk.MonthlyQuota}
}
s.cache.Set(cacheKey, res, ttlFor(res))          // write-through to both tiers
s.l2.Set(ctx, cacheKey, res, ttlFor(res))
Concept · the multi-replica staleness hole + pub/sub fix

Each replica holds its own L1, so a revoke on one replica would leave the others serving a stale "valid" for up to the TTL. Revoke does UPDATE Postgres → DELETE L2 → PUBLISH the key hash on a revocations channel; every replica's subscriber goroutine evicts that hash from its own L1 within milliseconds. TTL is the backstop if a message is missed.

keyservice/internal/cache/invalidation.go · fleet-wide eviction
func PublishRevocation(ctx context.Context, rdb *redis.Client, keyHash string) error {
	return rdb.Publish(ctx, RevocationChannel, keyHash).Err()
}
// SubscribeRevocations runs forever per replica, evicting each hash from local L1.
func SubscribeRevocations(ctx context.Context, rdb *redis.Client, l1 *Cache) {
	sub := rdb.Subscribe(ctx, RevocationChannel)
	defer sub.Close()
	for msg := range sub.Channel() {
		l1.Delete(msg.Payload)
	}
}

5.3 The gate — rate-limit + quota + meter in one atomic call

This is the heart of the hot path. A single Lua script — embedded into the binary and sent as a SHA via EVALSHA — refills a token bucket from elapsed time, then returns a 3-way verdict: 0 rate-limited, 2 quota-exceeded, or 1 allowed (spend a token, INCR usage). Only served requests are metered. There's no background timer — tokens accrue lazily per call, so idle buckets cost nothing and self-expire.

keyservice/internal/ratelimit/bucket.lua · the 3-way verdict
-- refill: add tokens for the time elapsed, capped at capacity
local elapsed = math.max(0, now - ts)
tokens = math.min(capacity, tokens + elapsed * rate)

local status
if tokens < 1 then
	status = 0                                       -- RATE_LIMITED
else
	local used = tonumber(redis.call("GET", usageKey)) or 0
	if quota > 0 and used >= quota then
		status = 2                                   -- QUOTA_EXCEEDED
	else
		tokens = tokens - 1
		local newused = redis.call("INCR", usageKey)
		if newused == 1 then
			redis.call("EXPIRE", usageKey, usage_ttl)
		end
		status = 1                                   -- ALLOWED + metered
	end
end
redis.call("HSET", bucketKey, "tokens", tokens, "ts", ts)
redis.call("EXPIRE", bucketKey, math.ceil(capacity / rate) + 1)  -- idle buckets self-clean
return status

The Go wrapper computes the per-month usage key and its TTL (time to month-end + 48 h grace, so the flusher can mirror the final value before Redis drops it), runs the script in one round-trip, and — critically — fails open: any Redis error returns Allowed.

keyservice/internal/ratelimit/ratelimit.go · Check — one round-trip, fails open
res, err := l.script.Run(ctx, rdb,
	[]string{"ratelimit:" + keyHash, usageKey},      // KEYS[1], KEYS[2]
	l.rate, l.capacity, nowSec, quota, usageTTL,     // ARGV[1..5]
).Int()
if err != nil {
	return Allowed                                   // fail OPEN — availability > strictness
}
switch res {
case 0:  return RateLimited
case 2:  return QuotaExceeded
default: return Allowed
}

5.4 The flusher — write-coalescing to Postgres

The hot path never writes to Postgres. Usage lives as Redis counters; a background flusher mirrors them on an interval. Only the replica that wins a self-expiring leader lease flushes that round — but correctness doesn't depend on the lease, because the UPSERT is idempotent.

keyservice/internal/usage/flusher.go · SCAN (never KEYS) → MGET → one batch
var keys []string
iter := f.rdb.Scan(ctx, 0, Prefix+"*", 100).Iterator()   // non-blocking cursor
for iter.Next(ctx) { keys = append(keys, iter.Val()) }
...
vals, err := f.rdb.MGet(ctx, keys...).Result()           // all counts, one round-trip
var rows []store.Usage
for i, key := range keys {
	if vals[i] == nil { continue }                       // key expired between SCAN and MGET
	period, tenantID, ok := ParseKey(key)
	count, _ := toInt64(vals[i])
	rows = append(rows, store.Usage{TenantID: tenantID, Period: period, Count: count})
}
return len(rows), f.store.UpsertUsage(ctx, rows)         // one batched UPSERT
keyservice/internal/store/store.go · idempotent absolute-value mirror
batch.Queue(
	`INSERT INTO usage_counters (tenant_id, period, count, updated_at)
	 VALUES ($1, $2, $3, now())
	 ON CONFLICT (tenant_id, period)
	 DO UPDATE SET count = EXCLUDED.count, updated_at = now()`,   // absolute, not delta
	u.TenantID, u.Period, u.Count,
)

Because the flusher writes the live total (not a delta), re-running is a no-op — which is what makes the leader lease an optimization (it suppresses duplicate work), not a correctness requirement.

5.5 Observability — RED without per-handler boilerplate

One middleware wraps the whole mux and records rate/errors/duration for every route uniformly. A statusRecorder captures the status code the stdlib otherwise swallows, and routeLabel uses the matched route template (r.Pattern) so /v1/tenants/{id}/usage is one Prometheus series, not one per UUID — the cardinality trap, avoided.

keyservice/internal/api/metrics.go · RED middleware + bounded labels
func metricsMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		rec := &statusRecorder{ResponseWriter: w, status: http.StatusOK}
		next.ServeHTTP(rec, r)
		route := routeLabel(r)                           // r.Pattern → bounded cardinality
		httpRequests.WithLabelValues(r.Method, route, strconv.Itoa(rec.status)).Inc()
		httpDuration.WithLabelValues(r.Method, route).Observe(time.Since(start).Seconds())
	})
}

Readiness (/readyz) pings Postgres + Redis under an 800 ms timeout and returns 503 to pull the replica from rotation — distinct from liveness (/healthz). Prometheus scrapes both replicas directly via dns_sd so their counters aren't blended. Logs are slog JSON, with the stdlib log package bridged through it.

6. The client SDK, the consumer & end-to-end tests

Concept · error ≠ verdict

The SDK's Validate returns an error only on transport failure (Mint unreachable). Mint's actual decision — allowed / invalid / rate-limited / quota-exceeded — rides back in an Outcome enum. That split is precisely what lets the middleware fail closed on an outage (503) while still mapping a genuine "invalid key" to 401.

keyservice-go/client.go · status → Outcome (stdlib-only module)
switch resp.StatusCode {
case http.StatusOK:
	return Result{Outcome: Allowed, TenantID: body.TenantID, KeyID: body.KeyID}, nil
case http.StatusUnauthorized, http.StatusBadRequest:
	return Result{Outcome: Invalid}, nil
case http.StatusTooManyRequests:
	if strings.Contains(body.Error, "quota") {           // 429 is ambiguous…
		return Result{Outcome: QuotaExceeded}, nil       // …body disambiguates
	}
	return Result{Outcome: RateLimited}, nil
default:
	return Result{}, fmt.Errorf("keyservice: unexpected status %d", resp.StatusCode)
}
keyservice-go/middleware.go · the one-line gate, fail-closed by default
res, err := c.Validate(r.Context(), key)
if err != nil {                                          // couldn't reach Mint
	if c.failOpen {
		next.ServeHTTP(w, r)                             // explicit opt-in
		return

	writeErr(w, http.StatusServiceUnavailable, "auth unavailable")   // fail closed

switch res.Outcome {
case Allowed:

	next.ServeHTTP(w, r.WithContext(ctx))                // inject identity, call handler
case RateLimited:   writeErr(w, http.StatusTooManyRequests, "rate limit exceeded")
case QuotaExceeded: writeErr(w, http.StatusTooManyRequests, "monthly quota exceeded")
default:            writeErr(w, http.StatusUnauthorized, "invalid api key")
}

A consumer gets all of that — auth, rate-limit, quota, identity injection — by wrapping its router once. The demo product (a stock-price API) is the proof: no security code in any handler.

7. Results

What was measured (reproducible from benchmarks/)

  • Cache before/after (hey -n 100000 -c 50): 12,018 → 18,292 rps (+52%); p99 17 → 10 ms (−41%); p50 3.2 → 2.2 ms. p99 is inside the 20 ms SLO.
  • Cache hit rate (Go loadgen, cold start, Zipf, 5% invalid): 99.7% @ 1k keys, 96.7% @ 10k keys over 300k requests — misses ≈ one per unique key.
  • Write reduction: ~10 Postgres writes/s = tenants / flush_interval, invariant to load. 376× end-to-end, 500× metering-isolated, durable mirror lossless at 600,000.
  • Image: ~22 MB multi-stage. Failure modes: auth fail-closed and rate-limit fail-open both verified by the integration suite.
  • Honesty: the ≥30k rps throughput target was not met on a shared laptop (18k real) — reported as-is; it needs dedicated hardware, and the before/after delta is the reliable signal.
