Chunk G, reconstructed bottom-up

Usage metering + monthly quota, from the design down to the lines — built the way you'd write it yourself. Every block answers “why does this exist?” before “what does it do?”, across three layers: design → logic → code.

System design — the why Logic — the how, no syntax Code — real lines Gotcha / tradeoff Margin note marker
The one-paragraph why Mint is a stateless Go service that other backends call on every request (POST /v1/validate) to check a key. Chunk G adds two things to that hot path: metering (count every served request, for billing) and quota (reject a tenant past its monthly cap). The hard constraint: at 5,000 rps you cannot write to Postgres per request. So G counts in Redis (atomic, in-memory) on the hot path and batch-mirrors the counts to Postgres in the background — the ~500× write-reduction target. Auth still fails closed; metering/quota fail open (availability > strictness).

Where Chunk G lives in the system

Green = added/changed in Chunk G. The validate handler now makes one Redis Lua call that gates and meters; a background goroutine drains the counters to Postgres.

The hot path: one request through the gate

Logic — control flow Validate the key from cache (RAM first, Redis second, Postgres last). If invalid → 401 and stop (never spend a token, never meter). If valid → make one Lua call that checks the rate-limit bucket, then the monthly quota, and only if both pass does it spend a token and INCR the usage counter. The verdict maps to 200 / 429-rate / 429-quota.

G.1 — schema + quota plumbing done

Design — the contract The quota check happens on the hot path, so it must be free to look up. The naive way — query the tenant's quota from Postgres per request — would defeat the entire cache. So the rule we lock in: the quota value rides inside the cached validation result. We add a column to tenants, read it in the same JOIN /validate already runs, and store it next to tenant_id in L1/L2. The quota then costs zero extra lookups. G.1 builds none of the enforcement — it just lays this pipe end-to-end.
Logic — the move
  • Schema: add tenants.monthly_quota (nullable; NULL = unlimited) and a usage_counters mirror table keyed (tenant_id, period).
  • One convention: map NULL → 0 with COALESCE on the hot-path read, so Go always sees a plain int64 where 0 = unlimited. (The admin response keeps a real *int64 so it can honestly show null.)
  • Carry it through: store.ValidateKey → cache.Result.MonthlyQuota → prewarm. Tenant creation accepts an optional quota.

Why Two durable facts must exist before any code: a place to store each tenant's cap, and a place to mirror the live Redis counters. NULL default = unlimited, because a metering service shouldn't impose a surprise cap.

2 store: read the quota on every path ▸
Why store owns all SQL — nothing above it writes a query. So every read that produces a cacheable result must now also surface the quota: the hot-path ValidateKey, the prewarm RecentActiveKeys, and the admin CreateTenant.

func (s *Store) ValidateKey(ctx context.Context, keyHash []byte) (ValidatedKey, error) {
	var vk ValidatedKey
	err := s.pool.QueryRow(ctx,
		`SELECT k.id, k.tenant_id, COALESCE(t.monthly_quota,0)     2
		FROM api_keys k
		JOIN tenants t ON t.id = k.tenant_id
		WHERE k.key_hash = $1 AND k.status = 'active' AND t.status = 'active'`,
		keyHash).Scan(&vk.KeyID, &vk.TenantID, &vk.MonthlyQuota)          3
	if err != nil {
		if errors.Is(err, pgx.ErrNoRows) {
			return ValidatedKey{}, ErrKeyNotValid       // fail closed + quiet
		}
		return ValidatedKey{}, err
	}
	return vk, nil
}

G.2 — the combined Lua call done

Design — the constraint that shapes everything The validate hot path already paid for one Redis round-trip in Chunk F (the token bucket). The locked-in decision: do not add a second round-trip for quota/metering — bundle all three into the one Lua script. Why Lua at all? Because rate-limit + quota + counter is a multi-step read-modify-write over shared state across replicas. Go locks can't coordinate two containers; Redis-side atomicity can. A Lua script runs atomically on Redis's single thread — no other command interleaves — so the whole gate is race-free in one shot.
Logic — the algorithm & two embedded decisions
  • Refill the bucket (lazy, time-based). If no token → return RATE_LIMITED, don't meter.
  • Else read the monthly counter. If used >= quota → return QUOTA_EXCEEDED, don't meter.
  • Else spend a token, INCR the counter, return ALLOWED.
Decision 1 — order: rate-limit before quota, so a hammered key (429-rate) doesn't burn the tenant's monthly allowance. Decision 2 — meter only on 200: usage = served requests; rejected ones (401/429) never increment. That's the correct billing semantic.
1 bucket.lua — the gate ▸
Why this file The script is the only place where the three checks can be made atomic and shared across replicas. It's go:embed-ed into the binary and sent to Redis as a SHA hash (EVALSHA) after first use — tiny hot-path payload, no separate file to deploy.
keyservice/internal/ratelimit/bucket.luaLua
-- KEYS[1]=bucket (per key)  KEYS[2]=usage counter (per tenant, per month)
-- ARGV: 1=rate 2=capacity 3=now(sec) 4=quota(<=0 unlimited) 5=usage TTL(sec)
local bucketKey = KEYS[1]
local usageKey  = KEYS[2]
local rate      = tonumber(ARGV[1])
local capacity  = tonumber(ARGV[2])
local now       = tonumber(ARGV[3])
local quota     = tonumber(ARGV[4])
local usage_ttl = tonumber(ARGV[5])

-- read + refill the token bucket
local state  = redis.call("HMGET", bucketKey, "tokens", "ts")     1
local tokens = tonumber(state[1])
local ts     = tonumber(state[2])
if tokens == nil then tokens = capacity; ts = now end     2
local elapsed = math.max(0, now - ts)
tokens = math.min(capacity, tokens + elapsed * rate)        3
ts = now

-- decide the verdict
local status
if tokens < 1 then
	status = 0                                              4
else
	local used = tonumber(redis.call("GET", usageKey)) or 0   5
	if quota > 0 and used >= quota then
		status = 2                                          6
	else
		tokens = tokens - 1
		local newused = redis.call("INCR", usageKey)         7
		if newused == 1 then redis.call("EXPIRE", usageKey, usage_ttl) end  8
		status = 1
	end
end

-- persist the bucket once (refill always; decrement only if allowed)
redis.call("HSET", bucketKey, "tokens", tokens, "ts", ts)   9
redis.call("EXPIRE", bucketKey, math.ceil(capacity / rate) + 1)
return status
1
HMGET = hash multi-get: read several fields of one hash in one call, returned in order. The bucket is a hash {tokens, ts} — two related numbers grouped under one key.
2
nil = first-ever request for this key → start the bucket full. (Reading a missing hash field returns nil; that's our "new bucket" signal.)
3
Lazy refill: tokens accrue from elapsed time, capped at capacity. No background timer, zero idle cost — the token-bucket algorithm.
4
No token → RATE_LIMITED. Note we fall through to the single bucket-write at the bottom (the refill must persist).
5
GET … or 0 — a never-seen counter is nil; treat as 0. We check before incrementing, so the boundary is exact (the (quota+1)th request is the first rejected).
6
quota > 0 guards the unlimited case (0 = unlimited). Over cap → QUOTA_EXCEEDED, and crucially no INCR — rejected requests aren't billed.
7
INCR = atomic +1 on a string-as-integer, returns the new value. This single command is the heart of metering: correct shared counting across both replicas with no Go locks.
8
Set the TTL only on the first INCR (newused == 1) — the key is born with "until month-end + grace" and we never reset it. Self-cleaning → bounded Redis memory.
9
One bucket write for every branch — DRY, and mirrors the original Chunk-F structure so the diff is minimal.
redis.call(cmd, key, args...) — Redis server-side Lua
does
Invokes a Redis command from inside the script and returns its reply as a Lua value. Runs in-process on Redis's single thread — nothing else interleaves for the whole script (atomic).
edge
Reading an undefined global (e.g. a typo'd variable name) raises a hard error and aborts the script — exactly the bug we hit with bucketKey vs key.
no rollback
If the script errors after a write (e.g. after INCR), Redis does not roll back. Combined with fail-open in Go, that produced the silent "always 200" symptom.
2 ratelimit.Check — Go side of the call ▸
Why Lua shouldn't do calendar math. Go computes the period string, the usage key, and the key's TTL, then hands them to the script. Allow (boolean) became Check (a 3-way Verdict). Still fails open — a Redis error returns Allowed (availability > strictness).
keyservice/internal/ratelimit/ratelimit.goGo
type Verdict int
const ( Allowed Verdict = iota; RateLimited; QuotaExceeded )   1

const usageGrace = 48 * time.Hour                              2

func (l *Limiter) Check(ctx context.Context, rdb *redis.Client,
                         keyHash, tenantID string, quota int64) Verdict {
	now := time.Now().UTC()                                  3
	period := now.Format("2006-01")                         // "2026-06"
	firstOfNext := time.Date(now.Year(), now.Month()+1, 1, 0,0,0,0, time.UTC)  4
	usageTTL := int(firstOfNext.Sub(now).Seconds()) + int(usageGrace.Seconds())
	usageKey := "usage:" + period + ":" + tenantID

	nowSec := float64(now.UnixNano()) / 1e9
	res, err := l.script.Run(ctx, rdb,
		[]string{"ratelimit:" + keyHash, usageKey},          5
		l.rate, l.capacity, nowSec, quota, usageTTL,            // ARGV[1..5]
	).Int()
	if err != nil { return Allowed }                          6
	switch res {                                              7
	case 0: return RateLimited
	case 2: return QuotaExceeded
	default: return Allowed
	}
}
1
iota auto-numbers a const block: Allowed=0, RateLimited=1, QuotaExceeded=2, all type Verdict. Idiomatic Go enum. These do not match the Lua numbers — the switch ⑦ is the bridge.
2
48h grace so the counter outlives month-end long enough for the flusher to mirror its final value before Redis drops it.
3
.UTC() — both replicas must agree on which month it is. Local time would split the period at a boundary.
4
time.Date(year, month+1, 1, …) — Go normalizes month 13 → next year's January, so end-of-month works even in December.
5
Two KEYS cross into Lua: the per-key bucket and the per-tenant counter. (Single Redis; in Cluster you'd hash-tag them to one slot.)
6
Fail open. Redis down → allow. The gate degrades to "no limit" rather than taking the whole service down.
7
Translate the Lua wire format (0/1/2) into the Go enum. Keeps the two numbering schemes decoupled.
script.Run(ctx, rdb, keys, args...).Int() — go-redis scripting
does
Executes the embedded Lua. First tries EVALSHA (send only the script's SHA); if Redis doesn't know it yet, falls back to EVAL (send full source) and caches it. So after the first call it's a tiny payload.
keys / args
[]string → KEYS[] in Lua; the variadic args → ARGV[]. go-redis stringifies numbers; Lua tonumbers them back.
.Int()
Coerces the reply to a Go int. Returns (int, error); a Redis/network error lands in err → fail open.
side effect
One network round-trip; mutates the bucket and (on allow) the counter atomically.

G.3a — the background flusher in progress

Design — write coalescing The counter is live in Redis but volatile (lost on restart). G.3a builds the goroutine that mirrors it to Postgres every ~30s, so it survives crashes and becomes queryable history. This is write coalescing: instead of one Postgres write per request (~5k/s), Postgres sees ~one write per tenant per interval — the ~500× reduction (Target #3). The mirror is Fork A: Redis holds the absolute monthly total; the flush overwrites Postgres with that value (idempotent), which is what lets the quota check stay a single cheap Redis read.
Logic — three moves
  • List safely: SCAN usage:* (cursor-based, non-blocking) — never KEYS.
  • Move in bulk: MGET all counts in one round-trip, then one batched UPSERT to Postgres.
  • Do it once: a Redis SET NX EX lease so only one replica flushes per round.

1 usage/keys.go — the key format ▸

keyservice/internal/usage/keys.goGo ✅ written
package usage

import "strings"

const Prefix = "usage:"                                   1

// ParseKey splits "usage:2026-06:<tenant>" into its period and tenant id.
func ParseKey(key string) (period, tenantID string, ok bool) {
	rest, found := strings.CutPrefix(key, Prefix)          2
	if !found { return "", "", false }
	return strings.Cut(rest, ":")                        3
}
1
One exported constant for the namespace — used both as the MATCH in SCAN usage:* and as the prefix to strip here.
2
strings.CutPrefix("usage:2026-06:abc", "usage:") → ("2026-06:abc", true). Returns found=false if the prefix isn't there → reject non-usage keys.
3
strings.Cut("2026-06:abc", ":") → ("2026-06", "abc", true). Splits at the first colon; safe because a UUID has none. We return its tuple straight as (period, tenantID, ok).
2 usage/flusher.go — the goroutine + lease ▸
Why One long-running goroutine per replica, ticking on an interval. It depends on two things across boundaries: *redis.Client (to read counters) and *store.Store (to write the mirror). Injecting both via NewFlusher keeps it testable — dependency injection.
keyservice/internal/usage/flusher.goGo · struct (✅ written)
const leaseKey = "lock:usage-flush"

type Flusher struct {
	rdb       *redis.Client     // reads usage counters + the lease
	store     *store.Store      // writes the durable mirror
	replicaID string            // stamped into the lease for observability
	interval  time.Duration
}

func NewFlusher(rdb *redis.Client, st *store.Store, replicaID string, interval time.Duration) *Flusher {
	return &Flusher{rdb: rdb, store: st, replicaID: replicaID, interval: interval}
}
How — the loop A time.Ticker fires every interval. Each tick: try the lease; if lost, skip; if won, flush once. select on ctx.Done() makes it cancellable.
keyservice/internal/usage/flusher.goGo · methods (⚠️ to write)
func (f *Flusher) Run(ctx context.Context) {
	ticker := time.NewTicker(f.interval)                  1
	defer ticker.Stop()
	for {
		select {
		case <-ctx.Done(): return                       2
		case <-ticker.C:
			if !f.acquireLease(ctx) { continue }         3
			n, err := f.flushOnce(ctx)
			if err != nil { log.Printf("usage flush failed: %v", err); continue }
			if n > 0 { log.Printf("usage flush: mirrored %d counter(s) (replica=%s)", n, f.replicaID) }
		}
	}
}

func (f *Flusher) acquireLease(ctx context.Context) bool {
	ok, err := f.rdb.SetNX(ctx, leaseKey, f.replicaID, f.interval).Result()  4
	if err != nil { return false }                       5
	return ok
}

func (f *Flusher) flushOnce(ctx context.Context) (int, error) {
	var keys []string
	iter := f.rdb.Scan(ctx, 0, Prefix+"*", 100).Iterator()       6
	for iter.Next(ctx) { keys = append(keys, iter.Val()) }
	if err := iter.Err(); err != nil { return 0, err }
	if len(keys) == 0 { return 0, nil }

	vals, err := f.rdb.MGet(ctx, keys...).Result()              7
	if err != nil { return 0, err }

	var rows []store.Usage
	for i, key := range keys {
		if vals[i] == nil { continue }                    8
		period, tenantID, ok := ParseKey(key)
		if !ok { continue }
		count, err := toInt64(vals[i])
		if err != nil { continue }
		rows = append(rows, store.Usage{TenantID: tenantID, Period: period, Count: count})  9
	}
	if err := f.store.UpsertUsage(ctx, rows); err != nil { return 0, err }  10
	return len(rows), nil
}

func toInt64(v any) (int64, error) {
	s, ok := v.(string)                                   11
	if !ok { return 0, fmt.Errorf("usage value not a string: %T", v) }
	return strconv.ParseInt(s, 10, 64)
}
1
time.NewTicker delivers a tick on channel ticker.C every interval. defer Stop() releases it so it doesn't leak.
2
<-ctx.Done() — when the context is cancelled (shutdown), this channel unblocks and we return cleanly. The select waits on whichever fires first.
3
Lost the lease → continue to the next tick. No flush this round on this replica.
4
The lease — see the API card below. Stamps replicaID as the value so you can see who's leading.
5
Redis error → return false (don't flush). Safe: the next round's winner catches up, and the UPSERT is idempotent anyway.
6
SCAN, never KEYS — cursor-based, non-blocking. The Iterator() hides the cursor loop. 100 is a per-step count hint, not a limit.
7
One MGET for all counts — one round-trip instead of N GETs. Result aligns with keys by index.
8
nil = key expired between SCAN and MGET (TTL). Skip it gracefully — a normal race, not an error.
9
Boundary: we build []store.Usage — the type the store package exposes for the write. That slice is the data crossing usage → store.
10
One call hands the whole batch to store. The flusher knows nothing about SQL; store knows nothing about Redis. Clean separation.
11
go-redis returns MGET values as any holding string. Type-assert, then strconv.ParseInt(s, base=10, bits=64) → int64.
rdb.SetNX(ctx, key, value, ttl).Result() — go-redis (the leader lease)
does
SET key value NX EX ttl in one atomic command: set only if the key does not exist (NX), with an auto-expiry (EX). This is a distributed lock / leader election primitive.
value
f.replicaID — who holds it (observability).
ttl
f.interval — the lock self-expires so a crashed leader can't block flushing forever.
returns
(bool, error): true = you won and may flush; false = someone else holds it. Error → treat as lost.
why safe
It's an optimization, not correctness: the UPSERT is idempotent, so even if two replicas flushed they'd write identical numbers.
rdb.Scan(ctx, cursor, match, count).Iterator() — go-redis
does
Incrementally walks the keyspace matching match, a chunk per round-trip, resuming from a server cursor. Non-blocking — Redis stays responsive between chunks. The opposite of KEYS, which scans everything in one blocking O(N) sweep.
cursor
0 = start. The Iterator() manages the cursor internally so you just Next() until done.
count
A hint for chunk size (100), not a cap on total results.
caveat
SCAN can return a key twice or miss keys created mid-scan — fine here (we re-flush every interval anyway).
3 store.UpsertUsage — the batched mirror ▸
Why store owns all SQL, so the write lives here, not in the flusher. It takes a slice and writes it as one pipelined batch — and writes the absolute value so re-running is a no-op (the idempotency the whole design leans on).
keyservice/internal/store/store.goGo (❌ to write)
type Usage struct { TenantID string; Period string; Count int64 }   1

func (s *Store) UpsertUsage(ctx context.Context, rows []Usage) error {
	if len(rows) == 0 { return nil }
	batch := &pgx.Batch{}                                  2
	for _, u := range rows {
		batch.Queue(
			`INSERT INTO usage_counters (tenant_id, period, count, updated_at)
			 VALUES ($1, $2, $3, now())
			 ON CONFLICT (tenant_id, period)
			 DO UPDATE SET count = EXCLUDED.count, updated_at = now()`,    3
			u.TenantID, u.Period, u.Count,
		)
	}
	br := s.pool.SendBatch(ctx, batch)                        4
	defer br.Close()
	for range rows {
		if _, err := br.Exec(); err != nil { return err }   5
	}
	return nil
}
1
The DTO that crosses usage → store. Plain fields, no SQL leaking up, no Redis leaking down.
2
pgx.Batch collects multiple statements to send together (pipelining) — one network exchange for N rows.
3
UPSERT. ON CONFLICT (tenant_id, period) DO UPDATE SET count = EXCLUDED.count — EXCLUDED is "the row I tried to insert". Overwrite (not +) = mirror the absolute Redis total = idempotent.
4
SendBatch ships the whole queue; defer Close() returns the connection to the pool.
5
Drain one result per queued statement; surface the first error. (Skipped: a deleted tenant would fail the FK — impossible today, no delete endpoint.)
pgx.Batch / pool.SendBatch / br.Exec — pgx pipelining
Batch.Queue
Appends a parameterized statement to the batch. No I/O yet — just buffering.
SendBatch
Sends every queued statement in one round-trip and returns a BatchResults to read replies from in order.
Exec()
Pulls the next statement's result; returns (CommandTag, error). Call it once per queued statement.
why
N tenants → 1 network exchange instead of N. With write-coalescing on top, Postgres load drops ~500×.

Cross-file map — who depends on whom

G.3b — the usage endpoint to write

Design — why this exists & the contract G.1–G.3a made usage flow (meter on the hot path, mirror to Postgres). But there's still no way to ask "how much has tenant X used this month, and how much is left?" over HTTP — you had to redis-cli GET and psql by hand. G.3b adds one admin endpoint that answers it. The key design idea is a two-source read that mirrors where the truth lives:
  • Live present → Redis. The current month's count is the live counter in Redis (usage:<period>:<tenant>) — always up to the second. The Postgres mirror lags by up to one flush interval, so for "right now" we read Redis.
  • Durable past → Postgres. Last month's number no longer exists in Redis (the key expired after its grace). Historical periods come from the usage_counters mirror. (G.3b builds the live/current path; the ?period=2026-05 historical variant is a noted extension.)
  • Quota → Postgres. The cap lives on tenants.monthly_quota; we read it fresh so the answer reflects any admin change immediately (the hot path caches it, but a stats endpoint should be exact).
The contract: GET /v1/tenants/{id}/usage (admin-authed) → 200 {tenant_id, period, used, quota, remaining}, where quota/remaining are null for an unlimited tenant. Bad UUID → 400, unknown tenant → 404.
Logic — the move
  • Auth (admin token), read {id} from the path.
  • Fetch quota from Postgres → map errors (invalid id → 400, missing → 404). This is also our existence check: no tenant row → 404 before we bother Redis.
  • Read the live used from Redis for the current period; a missing key (tenant made zero calls) just reads as 0 — fail-soft, a stats read should never 500 on a Redis miss.
  • Compute remaining = quota - used (clamped at 0); leave it null when unlimited.
One cleanup folded in: the usage-key format was being built inline in two places. G.3b centralizes it in the usage package (CurrentPeriod + Key) so the writer (ratelimit) and the readers (flusher, this endpoint) share one definition — no drift.

// GetTenantQuota returns a tenant's monthly cap (nil = unlimited).
func (s *Store) GetTenantQuota(ctx context.Context, tenantID string) (*int64, error) {
	var quota *int64                                          1
	err := s.pool.QueryRow(ctx,
		`SELECT monthly_quota FROM tenants WHERE id = $1`, tenantID,
	).Scan("a)
	if err != nil {
		if errors.Is(err, pgx.ErrNoRows) {
			return nil, ErrTenantNotFound                  2
		}
		var pgErr *pgconn.PgError
		if errors.As(err, &pgErr) && pgErr.Code == "22P02" {   3
			return nil, ErrInvalidTenantID
		}
		return nil, err
	}
	return quota, nil
}

var quota *int64 — scanning a nullable BIGINT into a *int64 leaves it nil on NULL, or points it at the value. That nil is "unlimited".
2

3

Gotchas & tradeoffs to remember

SCAN, not KEYS KEYS * blocks Redis (single-threaded, O(N) over the whole keyspace) — it freezes every other client. SCAN walks incrementally. KEYS is for a human at a console; SCAN is for code.
Mirror = absolute, not additive The UPSERT writes count = EXCLUDED.count. If it were count + EXCLUDED.count, every flush would re-add the running total and the number would balloon. Overwrite = idempotent = safe to double-flush.
The lease is an optimization Not a correctness mechanism. Because the UPSERT is idempotent, a missed or doubled flush is harmless; the lease just avoids wasted duplicate writes. So no Redlock-grade rigor needed.
Fail open vs fail closed Auth fails closed (no row → ErrKeyNotValid → 401). Rate-limit & quota fail open (Redis error → Allowed). Security is strict; the hot-path accelerators favor availability.
Known limits in G.3a The flusher re-mirrors every counter each round even if unchanged (wasteful, harmless — a "skip unchanged" optimization is a fair follow-up). A usage: key for a deleted tenant would fail the FK on UPSERT — impossible today (no delete-tenant endpoint).
Lua globals bite Reading an undefined global in a Redis Lua script raises and aborts. With Go failing open, that surfaced as "every request returns 200" — enforcement silently off. The fix was matching bucketKey top and bottom.

Glossary

Write coalescing
Batching many small writes into few. Here: per-request Redis INCR → one Postgres UPSERT per tenant per interval (~500×).
Token bucket
Rate-limit algorithm: a bucket refills at rate/s up to capacity; each request spends one token; empty → throttle. Allows bursts up to capacity.
UPSERT (ON CONFLICT)
Insert-or-update in one atomic statement. EXCLUDED = the row you tried to insert.
Idempotent
Running it again changes nothing further. Our mirror-UPSERT is idempotent → safe double-flush.
Leader lease / election
Coordinating "only one worker does X" via a short-lived lock (SET NX EX) that auto-expires.
Fail open / fail closed
On dependency failure, fail open = allow (availability), fail closed = deny (safety).
Dependency injection
Passing collaborators in (NewFlusher(rdb, st, …)) instead of constructing them inside — testable, swappable.
EVALSHA
Run a server-cached Lua script by its SHA hash, so the hot path sends a tiny identifier, not the source.
TTL grace
Extra lifetime (48h) on the counter past month-end, so the flusher captures the final value before Redis drops the key.
Volatile vs durable
Volatile = in-RAM, lost on restart (Redis). Durable = on-disk, survives crashes (Postgres).
