01What it is

The prices are fake and beside the point. Think of it as a placeholder for a real service that needs to charge and protect an API.

The demo is a tiny HTTP service — a stock-price API — living in its own Go module demo/. It imports the Mint client SDK (keyservice-go) the same way a real customer would: as an external dependency. Every route it exposes is gated by Mint, so an unknown key, a key over its rate limit, or a key past its monthly quota never reaches the handler.

That makes it the "customer's app" half of the picture. Mint is the central gate; the demo product is one of the many backends that consult it on every request — exactly how a backend calls Mint's /validate in production.

02How a request flows

A caller hits the demo product with a Bearer key. The product's middleware asks Mint to rule on it, then either serves the price or returns Mint's rejection.

The key idea: the demo product holds no secrets and no auth logic. It can't validate keys itself (that needs the pepper, Postgres, Redis) — it asks Mint over HTTP. The coupling is the /validate contract, nothing more.

03The endpoints

Two product routes plus an identity echo. All three sit behind the same one-line gate.

Method & pathReturnsPurpose
GET /v1/price/{symbol}{"symbol":"AAPL","usd":412.07,...}The "product" — a fake live price. The cheap endpoint the burst tool hammers.
GET /v1/whoami{"tenant":"...","key":"..."}Echoes who Mint authenticated — proves the tenant/key landed in the request context.

Behind the scenes Mint can answer four ways, and the middleware maps each to an HTTP status the caller sees:

SituationCaller gets
Valid key, within limits200 price served
Missing / unknown / revoked key401 invalid
Over rate limit, or over monthly quota429 too many requests
Mint unreachable503 fail-closed (no silent pass)

04The one-line integration

This is the whole point. The handlers are pure business logic; a single wrap makes every route authenticated, rate-limited, and metered.

demo/product/product.go
// Handler wires the routes and wraps them in Mint's auth middleware.
func Handler(client *keysvc.Client) http.Handler {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /v1/price/{symbol}", price)
	mux.HandleFunc("GET /v1/whoami", whoami)
	return client.Middleware(mux) // ← gates EVERY route above, one line
}

// Inside a handler, the caller's identity is just there — Mint put it
// in the context. No token parsing, no rate-limit check, no quota math.
func price(w http.ResponseWriter, r *http.Request) {
	tenant, _ := keysvc.TenantID(r.Context())
	// ... serve the price ...
}

Compare that to the alternative: every handler hand-rolling token extraction, a call to validate, rate-limit bookkeeping, quota accounting, and the right status codes. The middleware collapses ~40 lines of per-endpoint plumbing into one wrap — and new routes are protected automatically.

05Run the demo

Step 2 shows integration, Step 3 shows enforcement live, Step 4 shows it on the dashboards. Assumes the Mint stack is up (docker compose up -d).

  1. Start the demo service

    bash
    docker compose up -d --build demo
# or, without Docker, from the demo/ folder:
cd demo && go run .

  2. Provision a customer (tenant + API key)

    bash
    TID=$(curl -s -XPOST localhost:8080/admin/tenants \
  -H 'X-Admin-Token: just-works-for-now' -H 'content-type: application/json' \
  -d '{"name":"demo-co"}' | jq -r .id)
KEY=$(curl -s -XPOST localhost:8080/v1/tenants/$TID/keys \
  -H 'X-Admin-Token: just-works-for-now' -H 'content-type: application/json' \
  -d '{"name":"laptop"}' | jq -r .key)
echo "tenant=$TID  key=$KEY"

  3. The happy path & the gate

    bash
    # valid key → a price, and it knows who you are
curl -s localhost:9000/v1/price/AAPL -H "Authorization: Bearer $KEY"; echo
curl -s localhost:9000/v1/whoami     -H "Authorization: Bearer $KEY"; echo

# no key → blocked before the handler ever runs
curl -si localhost:9000/v1/price/AAPL | head -1
    200 price + your tenant id  ·  no key → 401

  4. Watch Mint enforce — the money shot

    bash · rate limit
    cd demo
# poll too fast → 200s turn into 429s
go run ./cmd/burst -key "$KEY" -n 2000 -c 50
    a few hundred allowed (the burst allowance), the rest rate_limited
    bash · quota
    # a free-tier customer capped at 50 calls total
TID2=$(curl -s -XPOST localhost:8080/admin/tenants \
  -H 'X-Admin-Token: just-works-for-now' -H 'content-type: application/json' \
  -d '{"name":"free-tier","monthly_quota":50}' | jq -r .id)
KEY2=$(curl -s -XPOST localhost:8080/v1/tenants/$TID2/keys \
  -H 'X-Admin-Token: just-works-for-now' -H 'content-type: application/json' \
  -d '{"name":"k"}' | jq -r .key)
go run ./cmd/burst -key "$KEY2" -n 300 -c 20
    allowed ≈ 50, then quota_exceeded climbs
    bash · stolen key
    go run ./cmd/burst -key "ak_live_garbage" -n 50 -c 10
    invalid = 50 — a fake key gets nothing

  5. See it on the dashboards

    While a burst runs, open Grafana at :3000 (no login). Service health shows the request wave, p99 latency, and rate/quota rejects ticking up; Tenants & quotas shows free-tier filling toward its 50-call cap.
