Chunk I — keyservice-go middleware, the plan

A tiny, drop-in Go package that other backends import so they can auth-check + rate-limit + quota every request through Mint with one line — instead of hand-rolling the /validate HTTP call. This is what makes Mint actually usable by a real product.

Decision — the why Plan — the how Target code Gotcha
The pain it removes Without this package, every consumer service repeats ~40 lines: pull the Authorization header → POST /v1/validate → parse JSON → map 200/401/429/400/5xx → allow or reject → stash the tenant id. Everyone gets the edge cases subtly wrong. Chunk I writes it once, correctly, behind:
consumer side
mux.Handle("/", keysvc.Middleware(myHandler))   // every request now gated by Mint

Where it sits in the system

The core decision: validate remotely, every request — no cache

Decision & defense The obvious optimization is to cache validate results and skip the network hop. We don't — and here's why: /validate does three things, and only one is cacheable.
What /validate checksCacheable?Why
Auth (is the key valid)yesvalidity changes only on revoke
Rate limitnoa per-request token-bucket decrement — caching "allowed" lets a key blow past its limit
Quotanoa per-request monthly counter — same problem
So Caching the result would break the centralized rate/quota guarantee — the entire reason Mint exists. So the middleware makes one /validate call per request. That's not a weakness: Mint's /validate is built for exactly this (L1+L2 cache + one Lua round-trip, ~5k rps, p99 ~10 ms). The consumer trades one fast hop for never implementing auth/rate/quota itself. (An auth-only cache is a documented future option — off by default, because it weakens the guarantee.)

Failure mode: fail closed by default, configurable

Decision If the consumer can't reach Mint, allow or deny? It's an auth gate, so the safe default is fail closed → 503 ("couldn't verify" — honest, not 401 "invalid"). Expose WithFailOpen() for consumers that prefer availability over strictness. (This mirrors Mint's own posture — auth fails closed, availability concerns fail open — but hands the choice to the consumer.)

Why an Outcome enum, not a bool The consumer can distinguish invalid vs rate-limited vs over-quota (different HTTP statuses, different retry advice), and can use Validate directly outside HTTP (gRPC, queue workers) — the Client is useful on its own, the Middleware is sugar over it.

Request flow through the middleware

Module layout — a separate, stdlib-only module

Why separate Consumers must go get a lightweight client — not drag in the server's pgx/redis deps. So keyservice-go is its own Go module at the repo root, stdlib-only (net/http, encoding/json, context). Zero third-party deps = trivial to adopt.

package keysvc

import (
	"context"
	"encoding/json"
	"fmt"
	"net/http"
	"strings"
	"time"
)

type Outcome int

const (
	Allowed Outcome = iota
	Invalid
	RateLimited
	QuotaExceeded
)

type Result struct {
	Outcome  Outcome
	TenantID string
	KeyID    string
}

type Client struct {
	baseURL  string
	http     *http.Client
	failOpen bool
}

type Option func(*Client)

func WithHTTPClient(h *http.Client) Option { return func(c *Client) { c.http = h } }
func WithTimeout(d time.Duration) Option   { return func(c *Client) { c.http.Timeout = d } }
func WithFailOpen() Option                 { return func(c *Client) { c.failOpen = true } }

func New(baseURL string, opts ...Option) *Client {
	c := &Client{
		baseURL: strings.TrimRight(baseURL, "/"),
		http:    &http.Client{Timeout: 2 * time.Second},
	}
	for _, o := range opts { o(c) }
	return c
}

// Validate calls Mint's POST /v1/validate. err != nil only on transport failure.
func (c *Client) Validate(ctx context.Context, key string) (Result, error) {
	req, err := http.NewRequestWithContext(ctx, http.MethodPost, c.baseURL+"/v1/validate", nil)
	if err != nil { return Result{}, err }
	req.Header.Set("Authorization", "Bearer "+key)               1

	resp, err := c.http.Do(req)
	if err != nil { return Result{}, err }                  2
	defer resp.Body.Close()

	var body struct {
		Valid    bool   `json:"valid"`
		TenantID string `json:"tenant_id"`
		KeyID    string `json:"key_id"`
		Error    string `json:"error"`
	}
	_ = json.NewDecoder(resp.Body).Decode(&body)

	switch resp.StatusCode {                                  3
	case http.StatusOK:
		return Result{Outcome: Allowed, TenantID: body.TenantID, KeyID: body.KeyID}, nil
	case http.StatusUnauthorized, http.StatusBadRequest:
		return Result{Outcome: Invalid}, nil
	case http.StatusTooManyRequests:
		if strings.Contains(body.Error, "quota") {           4
			return Result{Outcome: QuotaExceeded}, nil
		}
		return Result{Outcome: RateLimited}, nil
	default:
		return Result{}, fmt.Errorf("keyservice: unexpected status %d", resp.StatusCode)  5
	}
}
line notes ① forward the key to Mint as a Bearer — same scheme the consumer received it in. ② transport error → return it (the middleware turns this into 503/fail-open). ③ map Mint's status to an Outcome — the contract from Chunk D/F/G. ④ 429 carries the reason in the body; "quota" in the message ⇒ QuotaExceeded, else RateLimited. ⑤ 5xx etc. = "couldn't validate" → an error, distinct from a clean Invalid.
verify Tiny program (or test) calls New("http://localhost:8080").Validate(ctx, key) against running Mint → Allowed for a good key, Invalid for junk.

keyservice-go/middleware.goGo · target
package keysvc

import (
	"context"
	"encoding/json"
	"net/http"
	"strings"
)

type ctxKey int

const ( tenantKey ctxKey = iota; keyIDKey )      1

func (c *Client) Middleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		key, ok := bearer(r)
		if !ok {
			writeErr(w, http.StatusUnauthorized, "missing api key")
			return
		}
		res, err := c.Validate(r.Context(), key)
		if err != nil {                                  2
			if c.failOpen { next.ServeHTTP(w, r); return }
			writeErr(w, http.StatusServiceUnavailable, "auth unavailable")
			return
		}
		switch res.Outcome {                              3
		case Allowed:
			ctx := context.WithValue(r.Context(), tenantKey, res.TenantID)
			ctx = context.WithValue(ctx, keyIDKey, res.KeyID)
			next.ServeHTTP(w, r.WithContext(ctx))         4
		case RateLimited:
			writeErr(w, http.StatusTooManyRequests, "rate limit exceeded")
		case QuotaExceeded:
			writeErr(w, http.StatusTooManyRequests, "monthly quota exceeded")
		default:
			writeErr(w, http.StatusUnauthorized, "invalid api key")
		}
	})
}

func bearer(r *http.Request) (string, bool) {
	return strings.CutPrefix(r.Header.Get("Authorization"), "Bearer ")
}

// TenantID / KeyID let a handler read who the authenticated caller is.
func TenantID(ctx context.Context) (string, bool) { v, ok := ctx.Value(tenantKey).(string); return v, ok }
func KeyID(ctx context.Context) (string, bool)    { v, ok := ctx.Value(keyIDKey).(string); return v, ok }

func writeErr(w http.ResponseWriter, status int, msg string) {
	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(status)
	_ = json.NewEncoder(w).Encode(map[string]string{"error": msg})
}
line notes ① a private ctxKey type — using an unexported type as the context key prevents collisions with other packages' keys (the idiomatic Go pattern). ② Mint unreachable → fail closed (503) unless WithFailOpen(). ③ the Outcome → HTTP mapping, all in one place. ④ on success, attach tenant/key to the context and call the real handler with the enriched request.
verify Wrap a one-line handler that reads keysvc.TenantID(r.Context()); curl with a good key → 200 + tenant; junk → 401; over-quota key → 429.

Gotchas

Don't cache the result Auth is cacheable; rate-limit + quota are not (per-request state). Caching the validate result would let keys exceed limits — it breaks Mint's central guarantee. One call per request.
Fail closed by default Mint unreachable → 503, not "allow". It's an auth gate; opening it on a Mint blip is a security hole. WithFailOpen() is an explicit opt-in.
503 vs 401 on Mint error "Couldn't verify" (503) is not "key is invalid" (401). Returning 401 on an outage would lie to the caller and break their retry logic.
Unexported context key type Use a private ctxKey type, not a bare string, or you'll collide with other middleware writing to the same context.
Keep it stdlib-only No third-party deps in this module — the whole value is a trivial go get for consumers. Don't pull in the server's pgx/redis.

Glossary

Middleware
A wrapper around an http.Handler (func(next) http.Handler) that runs before/after the real handler — here, to authenticate.
Client library / SDK
A small package that wraps a remote API's HTTP calls so consumers don't hand-roll them.
Bearer token
The Authorization: Bearer <token> scheme; the middleware reads it from the incoming request and forwards it to Mint.
request context
Go's per-request key/value bag (context.Context); the middleware stashes tenant/key id there for the handler.
Fail open / fail closed
On a dependency failure, allow (availability) vs deny (security). This gate fails closed by default.
Outcome enum
Allowed / Invalid / RateLimited / QuotaExceeded — richer than a bool, so the consumer can react precisely.
Go module
A versioned unit of Go code with its own go.mod; a separate one keeps the client's dependency footprint tiny.
