Chunk H — Observability, the plan

Make the service visible: structured logs, a Prometheus /metrics endpoint, and a Grafana dashboard that turns every number in RESULTS.md into a live, presentable graph — plus a real readiness probe. This doc is the design pass (build-along phases 1–3); each piece answers why before how.

Decision — the why Plan — the how Target code Gotcha core H.1–H.2 optional H.3
The goal (and why it's its own chunk) Everything Mint does well — 18k rps, p99 10 ms, ~99.7% cache hit rate, ~10 writes/s from the flusher — is currently invisible unless you re-run a benchmark. Chunk H makes it continuously observable: the service emits metrics, Prometheus stores them, and Grafana draws them. The payoff is twofold — it's how you operate a service (spot a latency spike, see a dependency die) and, for a portfolio, it's the demo: a dashboard moving in real time as you hammer it.

The three pillars — and what we use for each

Plan Observability is conventionally split into three signal types. We invest where the payoff is for a single service: metrics (the dashboard + the perf-target story) first, logs second, traces optional.

How it flows — scrape & query

Plan The service is dumb: it just publishes current numbers at /metrics. Prometheus pulls (scrapes) that page every ~15 s and remembers it. Grafana queries Prometheus (PromQL) and draws graphs. Three moving parts, one direction of data.

H.1 — instrument the Go service core · code only

Plan — three additions, no new infra
  • A metrics middleware wrapping the router that records the RED signals for every request, plus an /metrics handler.
  • Promote the cache + flusher counters we already track into Prometheus metrics.
  • Switch log.Printf → slog (JSON), and add /readyz.

Metrics catalog (RED + the Mint-specific ones)

Why these RED = Rate, Errors, Duration — the canonical "what to measure" for a request-driven service. We add cache hit-rate, the flusher's write-rate (the live 500× story), and the 429 breakdown, so the dashboard mirrors every perf target.
MetricTypeLabelsAnswers
http_requests_totalCountermethod, route, statusRate + Errors (by route, by status)
http_request_duration_secondsHistogrammethod, routeDuration → p50/p95/p99 via histogram_quantile
validate_cache_events_totalCountertier = l1·l2·misscache hit rate (the 99.7% number)
validate_rejected_totalCounterreason = rate·quota429 breakdown (rate-limit vs quota)
usage_flush_writes_totalCounter—Postgres writes by the flusher (live write-rate ≈ 10/s)
usage_flush_duration_secondsHistogram—how long a flush takes
usage_flush_tenantsGauge—active tenants mirrored last flush
Free from client_golang: go_goroutines, process_resident_memory_bytes, GC stats, etc. — runtime health, no work.
Metric types (define inline)
  • Counter — only ever goes up (e.g. total requests). You graph its rate(), not the raw value.
  • Gauge — goes up and down (e.g. current goroutines, last-flush tenant count).
  • Histogram — samples bucketed by range, so Prometheus can compute quantiles (p99) server-side. The right type for latency.

The metrics middleware (RED)

Why a middleware RED must be recorded for every route uniformly. A middleware — a wrapper around the handler chain — is the idiomatic Go way: it starts a timer, calls the real handler, then records status + duration once. One place, every route.
keyservice/internal/api/metrics.goGo · target
var (
	httpReqs = promauto.NewCounterVec(prometheus.CounterOpts{
		Name: "http_requests_total", Help: "Total HTTP requests.",
	}, []string{"method", "route", "status"})

	httpDur = promauto.NewHistogramVec(prometheus.HistogramOpts{
		Name:    "http_request_duration_seconds", Help: "Request latency.",
		Buckets: []float64{.0005, .001, .0025, .005, .01, .025, .05, .1, .25, .5, 1},  // 0.5ms…1s
	}, []string{"method", "route"})
)

// statusRecorder captures the status code the handler writes (ResponseWriter hides it).
type statusRecorder struct {
	http.ResponseWriter
	status int
}
func (r *statusRecorder) WriteHeader(c int) { r.status = c; r.ResponseWriter.WriteHeader(c) }

// metricsMiddleware wraps the whole mux: time it, call it, record RED once.
func metricsMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		rec := &statusRecorder{ResponseWriter: w, status: 200}
		next.ServeHTTP(rec, r)
		route := routeLabel(r)                                  // TEMPLATE, not raw path — see gotcha
		httpReqs.WithLabelValues(r.Method, route, strconv.Itoa(rec.status)).Inc()
		httpDur.WithLabelValues(r.Method, route).Observe(time.Since(start).Seconds())
	})
}
🔌
Wiring: in Routes(), return metricsMiddleware(mux) instead of mux, and register mux.Handle("GET /metrics", promhttp.Handler()). The cache/flush metrics get .Inc() where those events already happen (the L1/L2/miss branches in handleValidate, the verdict switch, the flusher).

Structured logging (slog)

Plan Replace log.Printf with a JSON slog logger so logs are machine-parseable (and Loki-ready later). Set it once in main; level via env.
keyservice/main.goGo · target
// JSON structured logs to stdout; one logger for the whole service.
logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
	Level: slog.LevelInfo,
}))
slog.SetDefault(logger)
slog.Info("keyservice starting", "replica", replicaID, "addr", addr)
Gotcha — don't log every request At 5k rps, an info line per request is a firehose. Log errors and slow/non-2xx requests; let metrics carry the per-request aggregates. Logs are for "what exactly happened to this one," not for counting.

/readyz vs /healthz

Why two endpoints Liveness (/healthz) = "is the process up?" — already exists. Readiness (/readyz) = "can it actually serve?" — i.e. are Postgres and Redis reachable. A replica that's alive but whose Redis is down should be pulled from rotation (return 503) without being killed (liveness stays ok). That distinction is exactly what load balancers and Kubernetes act on.
keyservice/internal/api/api.goGo · target
func (s *Server) handleReadyz(w http.ResponseWriter, r *http.Request) {
	ctx, cancel := context.WithTimeout(r.Context(), 800*time.Millisecond)
	defer cancel()
	pgOK := s.store.Ping(ctx) == nil           // pool.Ping
	redisOK := s.rdb.Ping(ctx).Err() == nil
	code := http.StatusOK
	if !pgOK || !redisOK { code = http.StatusServiceUnavailable } // 503 → out of rotation
	w.WriteHeader(code)
	json.NewEncoder(w).Encode(map[string]bool{"ready": pgOK && redisOK, "postgres": pgOK, "redis": redisOK})
}

H.2 — Prometheus + Grafana core · the presentation

Plan Add two containers, point Prometheus at both keyservice replicas, and ship a pre-provisioned Grafana dashboard so it's live on docker compose up — no click-ops.

Scrape config — and the replica gotcha

Gotcha — scraping 2 replicas behind one name keyservice is one Compose service with replicas: 2 → one DNS name shared by two containers. Scraping keyservice:8080 directly would hit a random replica each time and blend their counters. Fix: Docker's embedded DNS returns all replica IPs for the service name, so use Prometheus DNS service discovery (dns_sd_configs, type A) to scrape every replica as its own target. Scrape directly on the internal network — not through nginx.
infra/prometheus.ymlYAML · target
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: keyservice
    dns_sd_configs:
      - names: ["keyservice"]      # Docker DNS returns all replica IPs
        type: A
        port: 8080
    metrics_path: /metrics
docker-compose.ymlYAML · target (additions)
  prometheus:
    image: prom/prometheus:v2.54.1
    volumes: ["./infra/prometheus.yml:/etc/prometheus/prometheus.yml:ro"]
    ports: ["9090:9090"]

  grafana:
    image: grafana/grafana:11.2.0
    ports: ["3000:3000"]
    environment:
      GF_AUTH_ANONYMOUS_ENABLED: "true"          # no login wall for the demo
      GF_AUTH_ANONYMOUS_ORG_ROLE: "Admin"
    volumes:
      - "./infra/grafana/provisioning:/etc/grafana/provisioning:ro"
      - "./infra/grafana/dashboards:/var/lib/grafana/dashboards:ro"
🗂️
Grafana provisioning = drop a datasource YAML (points at http://prometheus:9090) and a dashboard JSON into mounted folders; Grafana loads them on startup. That's what makes the dashboard appear automatically — reproducible, version-controlled, no manual setup.

Gotchas to respect

Label cardinality Every unique label-value combo is a separate time-series. Never label by key hash, tenant_id, or raw path — that's unbounded and will blow up Prometheus's memory. Use the route template (/v1/tenants/{id}/keys) and a bounded status set.
Scrape each replica (repeated because it bites) — scrape replicas directly via dns_sd, not keyservice:8080 round-robined through one name, or you'll blend two replicas' counters.
Log volume No info-log per request at 5k rps. Errors + slow/non-2xx only; metrics carry the aggregates.
Histogram buckets Quantiles are only as good as the buckets. Latency lives in the 0.5 ms–25 ms range here, so bucket densely there (see the middleware) — default buckets (5 ms–10 s) would make p99 useless.
/metrics is internal Don't expose /metrics through nginx to the public — it leaks operational detail. Scrape it on the internal Docker network only.

Glossary

Observability
Being able to ask, from the outside, what a system is doing inside — via metrics, logs, traces.
Prometheus
Time-series database that periodically scrapes a service's /metrics endpoint and stores the numbers; queried with PromQL.
Grafana
Dashboard tool that draws graphs by querying a datasource (Prometheus here). Stores nothing itself.
Scrape (pull model)
Prometheus fetching /metrics on an interval. Opposite of push; lets Prometheus detect a down target.
Time-series
A stream of (timestamp, value) points for one metric+label-set.
PromQL
Prometheus's query language — rate(), histogram_quantile(), aggregations by label.
Counter / Gauge / Histogram
Up-only / up-and-down / bucketed-for-quantiles metric types.
RED method
Rate, Errors, Duration — the standard signals for a request-driven service. (USE = Utilization/Saturation/Errors, for resources.)
SLI / SLO
SLI = the measured indicator (p99 latency). SLO = the target on it (p99 < 20 ms). The gauge shows current vs target.
Cardinality
Number of distinct time-series = product of label-value counts. High cardinality (per-id labels) is the classic Prometheus footgun.
Liveness vs readiness
Liveness = process up (/healthz). Readiness = can serve, deps reachable (/readyz). LBs route on readiness, restart on liveness.
slog
Go 1.21+ standard-library structured logger (JSON/text handlers, key-value attributes).
Provisioning (Grafana)
Declaring datasources + dashboards as files Grafana loads on boot — reproducible, no click-ops.
OpenTelemetry / Tempo
Vendor-neutral instrumentation SDK (OTel) + a traces backend (Tempo). The optional H.3 tracing path.
